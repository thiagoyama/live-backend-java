conceito fundamental

🤔 O Que É Uma API?

🔌
API = Application Programming Interface

É um contrato de comunicação entre sistemas. Define como uma aplicação pode pedir e receber dados de outra — sem precisar saber como ela funciona por dentro.

🍽️
A Analogia do Restaurante

Você = Frontend / App Mobile
Garçom = API REST
Cozinha = Backend / Banco de Dados

Você pede ao garçom. O garçom leva à cozinha e traz de volta. Você não precisa entrar na cozinha!

🌍
APIs no Mundo Real

💳 Pagamento com cartão → API de pagamento
📍 Google Maps em apps → API do Maps
🔑 "Entrar com Google" → API OAuth
🎮 Steam verifica sua biblioteca → API da Steam
→ Tudo que conecta sistemas é uma API!

💡 A nossa API de Games vai permitir que qualquer app (web, mobile, outro servidor) liste, cadastre, atualize e delete games via HTTP.

CLIENTE (Frontend)

Faz requisições HTTP para consumir recursos. Não sabe como o backend funciona — só conhece a URL e o formato do JSON.

SERVIDOR (Backend)

Recebe requisições, processa a lógica de negócio, acessa o banco e devolve uma resposta em JSON com um status code.

SEPARAÇÃO DE RESPONSABILIDADES

Frontend e backend evoluem independentemente. Um app mobile e um web podem consumir a mesma API!

protocolo de comunicação

🌐 O Protocolo HTTP

📡
HTTP — HyperText Transfer Protocol

Define como as mensagens são trocadas entre cliente e servidor na web. É a linguagem universal da internet — toda requisição e resposta seguem esse protocolo.

🔄
Ciclo Request / Response

1️⃣ Request: cliente envia verbo + URL + body (se necessário)
2️⃣ Servidor processa e acessa o banco
3️⃣ Response: servidor responde com status code + body JSON

💡 Quando você abre localhost:8080/api/games no navegador, você está fazendo uma requisição HTTP GET para o seu backend Spring Boot!

VERBO OPERAÇÃO CRUD EXEMPLO — GAMES API O QUE FAZ BODY?
GET READ GET /api/games Busca/lista recursos — só lê, nunca altera Não

respostas do servidor

📊 HTTP Status Codes — O Servidor Fala

200
OK
Requisição bem-sucedida. Usado em GET e PUT.
201
CREATED
Recurso criado com sucesso. Retornado pelo POST.
204
NO CONTENT
Operação ok, sem corpo na resposta. Usado no DELETE.
400
BAD REQUEST
Erro na requisição do cliente (dados inválidos, JSON malformado).
404
NOT FOUND
Game não encontrado no banco. ID inexistente.
500
SERVER ERROR
Erro interno no servidor. Problema no backend ou banco.
2xx = ✅ Sucesso

Tudo correu bem. O servidor processou e respondeu.

4xx = ⚠️ Erro do cliente

Problema na requisição enviada. Corrija e tente novamente.

5xx = 💥 Erro do servidor

O backend explodiu. Problema a investigar no log do Spring.

formato de dados

📦 JSON — A Linguagem da API

📝
JSON — JavaScript Object Notation

Formato padrão para troca de dados em APIs REST. Estrutura de chave–valor, leve e legível por humanos e máquinas.

🔀
Java ↔ JSON (Jackson)

O Spring Boot usa a biblioteca Jackson para converter automaticamente:

📦 Objeto Java Game → JSON (resposta)
📥 JSON do body → Objeto Java Game (requisição)

🎮 Você escreve Java. O Spring converte para JSON automaticamente. Sem código extra!

EXEMPLO — GAME COMO JSON

🏛️ REST e Maturidade de Richardson

0
Pântano de XML
1 endpoint para tudo. Verbo errado. XML como formato. Não é REST.
❌ Evitar
1
Recursos
URLs separadas por recurso: /games, /users. Ainda sem verbos corretos.
⚠️ Básico
2
Verbos HTTP ★
GET/POST/PUT/DELETE corretos + status codes corretos. Isso é o mínimo para ser RESTful!
✅ Nossa meta
3
HATEOAS
Resposta inclui links de navegação entre recursos. API "autodescoberta".
🚀 Avançado
🎯 Implementar o Nível 2 = sua API é considerada RESTful. É exatamente o que vamos construir com Spring Boot hoje!
💡 Roy Fielding (2000) e Leonard Richardson (2008) definiram esses padrões. Todo back-end Java do mercado segue o Nível 2 no mínimo.

O QUE CADA OPÇÃO DE ddl-auto FAZ?

none JPA não mexe no banco. Você cria e gerencia as tabelas manualmente pelo SQL Developer. ← nossa escolha!
validate JPA só confere se a tabela bate com a entidade. Erro se divergir. Não altera nada.
update JPA cria colunas novas se necessário, mas nunca remove. Útil em dev, perigoso em produção.
create Dropa e recria a tabela ao subir. Apaga todos os dados! Só em dev/testes.
create-drop Cria ao subir e dropa ao fechar. Apenas para testes automatizados.
⚠️ Como a tabela tb_game já existe no banco Oracle FIAP, usamos none para não mexer no que já está lá!
passo 3 de 6 · model

📦 Entidade JPA — Game.java

@Entity // "isso é uma tabela JPA" @Table(name = "TB_GAME") // nome real da tabela Oracle public class Game { @Id @Column(name = "cd_game") @SequenceGenerator(name = "game", sequenceName = "sq_game", allocationSize = 1) @GeneratedValue(generator = "game", strategy = GenerationType.SEQUENCE) private long codigo; // mesma sequence da live 01! @Column(name = "nm_game") @NotBlank @Size(max = 100, min = 3) private String nome; @Column(name = "nr_nota") @DecimalMin(value = "0", message = "Nota mínima é 0") @DecimalMax(value = "10", message = "Nota máxima é 10") private double nota; @Column(name = "nr_ano") private int ano; @Column(name = "dt_compra") private LocalDate dataCompra; // DATE → LocalDate @Column(name = "st_finalizado") private boolean finalizado; // NUMBER(1) → boolean // getters e setters... }

PRINCIPAIS ANOTAÇÕES JPA

@Entity
Marca a classe como entidade JPA — o Hibernate vai gerenciá-la e mapeá-la para uma tabela.
@Table(name="TB_GAME")
Define o nome real da tabela no banco. Se omitir, usa o nome da classe em minúsculo.
@Id + @SequenceGenerator + @GeneratedValue
Define a chave primária e usa a sequence sq_game do Oracle para gerar o código automaticamente.
@Column(name="nm_game")
Mapeia o atributo Java para a coluna exata do banco. Necessário quando o nome do atributo difere da coluna.
@NotBlank · @Size · @DecimalMin · @DecimalMax
Bean Validation — valida os dados recebidos. O @Valid no Controller ativa essas regras automaticamente.

passo 5 de 6 · service

⚙️ Por que criar o GameService?

🏗️
Separação de Responsabilidades

O Controller cuida apenas de receber a requisição HTTP e devolver a resposta JSON.

O Service é onde colocamos a lógica de negócio — regras, validações e decisões que não têm nada a ver com HTTP.

✅
O que implementamos no Service

🔍 Verificar existência antes de atualizar/deletar
⚠️ Lançar exceção se game não existir → HTTP 404
🛡️ Optional<T> para evitar NullPointerException
🏷️ Garantir o ID correto no update (game.setCdGame(id))

@Service public class GameService { @Autowired private GameRepository repository; public Game buscarPorId(Long id) { Optional<Game> opt = repository.findById(id); if (opt.isPresent()) { return opt.get(); } else { throw new GameNaoEncontradoException(id); // Controller captura → devolve 404 } } public Game atualizar(Long id, Game game) { Optional<Game> existente = repository.findById(id); if (existente.isPresent()) { game.setCdGame(id); // garante ID correto return repository.save(game); } else { throw new GameNaoEncontradoException(id); } } public List<Game> listarTodos() { return repository.findAll(); } // salvar(), excluir()... }
🎮 Optional<T> é um "container" que pode ou não ter valor. .isPresent() verifica se achou algo — evita o famoso NullPointerException em produção!
passo 6 de 6 · controller

🌐 GameController — Definindo a API

@RestController // responde JSON automaticamente @RequestMapping("/api/games") // URL base de todos os endpoints public class GameController { @Autowired private GameService service; @GetMapping @ResponseStatus(HttpStatus.OK) // 200 public List<Game> listarTodos() { return service.listarTodos(); } @PostMapping @ResponseStatus(HttpStatus.CREATED) // 201 public Game salvar(@RequestBody @Valid Game game) { return service.salvar(game); } @GetMapping("/{id}") public ResponseEntity<Object> buscarPorId(@PathVariable Long id) { try { return ResponseEntity.ok(service.buscarPorId(id)); } catch (GameNaoEncontradoException e) { return ResponseEntity.status(404).body(e.getMessage()); } } @PutMapping("/{id}") public ResponseEntity<Object> atualizar( @PathVariable Long id, @RequestBody @Valid Game game) { try { return ResponseEntity.ok(service.atualizar(id, game)); } catch (GameNaoEncontradoException e) { return ResponseEntity.status(404).body(e.getMessage()); } } @DeleteMapping("/{id}") public ResponseEntity<Object> excluir(@PathVariable Long id) { try { service.excluir(id); return ResponseEntity.noContent().build(); // 204 } catch (GameNaoEncontradoException e) { return ResponseEntity.status(404).body(e.getMessage()); } } }

ANOTAÇÕES DO CONTROLLER

@RestController
= @Controller + @ResponseBody. Todo retorno é serializado para JSON pelo Jackson.
@RequestMapping / @GetMapping / @PostMapping...
Define a URL e o verbo HTTP que ativa cada método.
@RequestBody
Jackson lê o JSON do body da requisição e converte para o objeto Java Game.
@PathVariable
Extrai o {id} da URL: GET /api/games/42 → id = 42L.
@ResponseStatus + ResponseEntity
Controla o status HTTP devolvido: 200, 201, 204, 404...
